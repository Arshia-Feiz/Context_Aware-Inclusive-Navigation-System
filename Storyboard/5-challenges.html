Key Challenges & Solutions
Challenge
Multi-Format Data Integration
Merging 5 disparate data sources with different coordinate systems, schemas, and encodings into a unified queryable format was the biggest technical hurdle.
Solution
Built a unified ContextDataLoader with format-specific parsers (Commons CSV, Jackson GeoJSON). Robust error handling skips corrupted rows without failing the entire load.
Challenge
Real-Time Query Performance
The A* algorithm queries the database for every edge evaluation. With thousands of edges per route calculation, naive queries caused unacceptable latency.
Solution
Spatial bounding box queries with indexed lat/lng columns. Batch insert optimizations (1000 records) and H2 file-mode persistence avoid reload overhead.
Challenge
Incomplete Data Coverage
Not all streets in Toronto have crime stats, lighting data, or surface information. Missing data could produce inaccurate or broken routes.
Solution
Graceful degradation: when context data is unavailable for a segment, the system falls back to distance-based routing. Default neutral scores prevent bias.

Limited Data for This Area
Street lighting and crime data is unavailable for the selected route segment. The system will calculate a standard distance-based route instead.
Lesson Learned: Designing for incomplete data from the start was critical. Instead of blocking the user when context data is missing, the application gracefully degrades to standard routing. This ensures the app remains fully functional even with gaps in the Toronto Open Data coverage.
